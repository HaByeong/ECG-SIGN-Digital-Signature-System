## 1. 프로젝트 개요

### 30초 버전
"ECG 심전도 신호의 고유한 파형 특성을 이용한 생체 인증 시스템입니다. 단순한 심박수가 아닌, P파, QRS 복합체, T파 등 개인마다 고유한 파형 정보를 추출하여 디지털 서명으로 변환하고, 이를 사용자 인증에 활용합니다."

### 1분 버전
"ECG-SIGN은 심전도 신호를 기반으로 한 디지털 서명 인증 시스템입니다. 

**핵심 아이디어**: 사람마다 ECG 파형이 다르다는 점을 활용했습니다. 단순히 심박수만 보는 것이 아니라, P파, QRS 복합체, T파의 진폭, 지속 시간, 간격 등 **형태학적 특징**을 추출하여 개인을 식별합니다.

**구현 내용**: Arduino와 AD8232 센서로 ECG 데이터를 수집하고, Android 앱을 통해 실시간으로 표시하며, Python 서버에서 Pan-Tompkins 알고리즘으로 R-peak를 검출하고 특징을 추출한 후 SHA-256 해시 기반 서명을 생성합니다. 사용자 등록 시 서명을 저장하고, 로그인 시 비교하여 인증합니다."

### 2분 버전 (상세)
"ECG-SIGN은 ECG 심전도 신호를 이용한 생체 인증 시스템입니다.

**프로젝트 배경**: 기존 생체 인증(지문, 얼굴 인식)의 한계를 보완하고, ECG의 고유성을 활용한 새로운 인증 방식을 연구하고자 시작했습니다.

**시스템 구성**:
1. **하드웨어**: Arduino + AD8232 ECG 센서로 500Hz 샘플링
2. **Android 앱**: Bluetooth로 데이터 수신, 실시간 그래프 표시, TCP로 서버에 전송
3. **Python 서버**: ECG 신호 처리 파이프라인, 사용자 인증 관리

**처리 과정**:
- 전처리: 노이즈 제거 (baseline wander, 고주파 노이즈, 60Hz 전원 노이즈)
- R-peak 검출: Pan-Tompkins 알고리즘 사용
- 특징 추출: 형태학적(진폭, 간격), 시간 영역(HRV), 주파수 영역(FFT), 통계적 특징
- 서명 생성: 특징 벡터화 → 정규화 → 이산화 → SHA-256 해시

**결과**: 사용자 등록 후 로그인 시 ECG 서명을 비교하여 인증합니다."

## 2. 핵심 기술 스택 및 선택 이유

### 하드웨어
- **Arduino + AD8232**: 
  - **선택 이유**: ECG 센서 중 가장 널리 사용되고, 가격이 저렴하며, Arduino와의 호환성이 좋습니다.
  - **역할**: 아날로그 ECG 신호를 디지털로 변환 (ADC), 500Hz 샘플링

- **HC-06 Bluetooth 모듈**:
  - **선택 이유**: Bluetooth Classic(SPP)을 지원하여 간단한 시리얼 통신으로 데이터 전송이 가능합니다.
  - **역할**: Arduino에서 Android로 무선 데이터 전송

### Android
- **Java**:
  - **선택 이유**: Android 개발의 표준 언어이며, Bluetooth 및 네트워크 API가 잘 지원됩니다.
  
- **MPAndroidChart**:
  - **선택 이유**: 실시간 ECG 파형을 그래프로 표시하기 위해 오픈소스 라이브러리를 사용했습니다.

- **Bluetooth Classic (SPP)**:
  - **선택 이유**: HC-06 모듈과 호환되며, 단순한 시리얼 통신으로 구현이 간단합니다.

- **TCP Socket**:
  - **선택 이유**: Python 서버와의 실시간 양방향 통신이 필요했고, HTTP보다 낮은 레이턴시가 필요했습니다.

### Python 서버
- **Python**:
  - **선택 이유**: NumPy, SciPy 등 신호 처리 라이브러리가 풍부하고, 프로토타이핑이 빠릅니다.

- **NumPy**:
  - **선택 이유**: 대용량 배열 연산과 수치 계산에 최적화되어 있습니다.

- **SciPy**:
  - **선택 이유**: Butterworth 필터, FFT 등 신호 처리 함수가 구현되어 있습니다.

- **Socket (TCP)**:
  - **선택 이유**: 실시간 스트리밍 데이터 수신에 적합하며, 멀티스레딩으로 여러 클라이언트를 동시에 처리할 수 있습니다.

### 알고리즘
- **Pan-Tompkins 알고리즘**:
  - **선택 이유**: ECG R-peak 검출의 표준 알고리즘이며, 실시간 처리에 적합하고 정확도가 높습니다.
  - **특징**: 밴드패스 필터링, 미분, 제곱, 적분, 적응형 임계값을 사용합니다.

- **SHA-256 해시**:
  - **선택 이유**: 디지털 서명의 무결성을 보장하고, 일방향 함수로 원본 데이터를 복원할 수 없어 보안성이 높습니다.

## 3. 시스템 아키텍처 설명

### 전체 구조
```
[Arduino + AD8232] 
    ↓ Bluetooth (SPP, 500Hz)
[Android App]
    ↓ TCP/IP Socket
[Python Server]
    ↓ ECG Processing Pipeline
[ECG Signature]
```

### 데이터 흐름

**1. 데이터 수집 단계**
- Arduino: AD8232 센서에서 아날로그 값 읽기 (A0 핀)
- 500Hz 샘플링 (2ms 간격)
- Bluetooth로 한 줄에 하나의 정수 전송 (예: `512\n`)

**2. 데이터 전송 단계**
- Android: Bluetooth로 데이터 수신
- 실시간 그래프 표시 (MPAndroidChart)
- TCP Socket으로 Python 서버에 전송

**3. 데이터 처리 단계**
- Python 서버: 10000개 샘플 버퍼링 (약 20초)
- ECG 처리 파이프라인 실행:
  1. 전처리 (노이즈 제거)
  2. R-peak 검출 (Pan-Tompkins)
  3. 비트 분할 및 템플릿 생성
  4. 특징 추출
  5. 서명 생성 (SHA-256)

**4. 인증 단계**
- 등록: ECG 서명을 JSON 파일에 저장
- 로그인: 입력된 ECG 서명과 저장된 서명 비교 (코사인 유사도)
- 임계값(0.75) 이상이면 인증 성공

### 통신 프로토콜

**Bluetooth (Arduino → Android)**
- 프로토콜: Bluetooth Classic SPP
- 데이터 형식: ASCII 문자열 (한 줄에 하나의 정수)
- 샘플링: 500Hz

**TCP/IP (Android → Python)**
- 프로토콜: TCP Socket
- 데이터 형식: 
  - ECG 데이터: 정수 문자열 (한 줄에 하나)
  - 명령어: `CMD:REGISTER:<user_id>`, `CMD:LOGIN`, `CMD:LOGOUT` 등
- 응답: JSON 형식

## 4. 핵심 알고리즘 및 구현

### 4.1 Pan-Tompkins 알고리즘 (R-peak 검출)

**왜 필요한가?**
- ECG 신호에서 R-peak를 정확히 찾아야 심박수 계산 및 비트 분할이 가능합니다.
- R-peak는 QRS 복합체의 최고점으로, 가장 뚜렷한 특징입니다.

**알고리즘 단계**:

1. **밴드패스 필터 (5-15Hz)**
   - QRS 복합체의 주파수 대역을 강조
   - 다른 파형(P, T파)과 노이즈를 억제

2. **미분 필터**
   - 신호의 기울기를 강조
   - 급격한 변화(QRS)를 더 뚜렷하게 만듦
   - 공식: `H(z) = (1/8T)(-z^-2 - 2z^-1 + 2z + z^2)`

3. **제곱**
   - 모든 값을 양수로 변환
   - 큰 변화를 더욱 증폭

4. **이동 평균 (150ms 윈도우)**
   - 적분 효과로 신호를 평활화
   - 피크 검출을 용이하게 함

5. **적응형 임계값 피크 검출**
   - 평균 + 0.5 * 표준편차를 초기 임계값으로 설정
   - RR 간격 제약 조건 적용 (200ms ~ 2000ms)

6. **R-peak 정제**
   - 적분 신호의 피크는 약간 지연될 수 있음
   - 원본 신호에서 ±50ms 범위 내 최대값을 찾아 정확한 위치 결정

**구현 코드 위치**: `Software/Signal_Processing/ecg_processor/r_peak_detector.py`

### 4.2 특징 추출

**4가지 특징 카테고리**:

1. **형태학적 특징** (가중치: 1.5)
   - P, Q, R, S, T 파의 진폭
   - QRS 지속 시간, PR 간격, QT 간격
   - ST 세그먼트 높이
   - 파형 비율 (P/R, T/R)
   - 기울기 (R 상승/하강)
   - 면적 (QRS, P, T)

2. **시간 영역 특징 (HRV)** (가중치: 1.0)
   - 평균 RR 간격
   - SDNN (표준편차)
   - RMSSD (제곱평균제곱근 차이)
   - pNN50 (50ms 이상 차이 비율)
   - 변동계수

3. **주파수 영역 특징** (가중치: 0.8)
   - FFT 변환
   - LF, MF, HF 파워
   - LF/HF 비율
   - 스펙트럼 중심, 확산
   - 주요 주파수
   - 상위 FFT 계수

4. **통계적 특징** (가중치: 0.7)
   - 평균, 표준편차
   - 왜도 (skewness)
   - 첨도 (kurtosis)
   - 에너지, RMS
   - 제로 크로싱 비율
   - 엔트로피

**가중치를 적용한 이유**: 형태학적 특징이 개인 식별에 가장 중요하므로 높은 가중치를 부여했습니다.

### 4.3 서명 생성

**과정**:

1. **특징 벡터화**
   - 모든 특징을 하나의 벡터로 결합
   - 가중치 적용

2. **정규화 (Min-Max)**
   - 모든 값을 0-1 범위로 변환
   - 특징 간 스케일 차이 제거

3. **이산화**
   - 연속 값을 0-255 정수로 변환 (8비트)
   - 노이즈에 강건하게 만듦

4. **SHA-256 해시**
   - 이산화된 벡터를 바이트로 변환
   - SHA-256 해시 계산
   - 64자 16진수 문자열로 변환

**왜 해시를 사용하는가?**
- 원본 ECG 데이터를 저장하지 않아 프라이버시 보호
- 일방향 함수로 복원 불가능
- 고정 길이 출력으로 저장 효율적

### 4.4 인증 (서명 비교)

**방법**: 코사인 유사도 (Cosine Similarity)

```
similarity = (A · B) / (||A|| × ||B||)
```

- A: 등록 시 생성된 정규화된 특징 벡터
- B: 로그인 시 생성된 정규화된 특징 벡터
- 결과: 0-1 사이 값 (1에 가까울수록 유사)

**임계값**: 0.75
- 이 값 이상이면 동일인으로 판단
- 실제 데이터로 최적값을 찾아야 함 (현재는 실험적 값)

## 5. 구현 과정의 도전과 해결

### 도전 1: Bluetooth 연결 불안정

**문제**: HC-06 모듈과 Android 앱 간 연결이 자주 끊김

**해결**:
- 연결 실패 시 재시도 로직 추가
- 예외 처리 강화 (IOException 처리)
- 연결 상태를 UI에 실시간 표시

**학습**: 하드웨어 통신에서는 예외 처리가 중요하며, 사용자에게 명확한 피드백을 제공해야 함

### 도전 2: 실시간 데이터 처리

**문제**: 500Hz 샘플링으로 초당 500개 데이터가 들어오는데, UI 업데이트가 느려서 그래프가 끊김

**해결**:
- 그래프 데이터를 최대 500개 포인트로 제한 (슬라이딩 윈도우)
- UI 업데이트는 Handler를 통해 메인 스레드에서 수행
- 데이터 수신은 별도 스레드(ConnectedThread)에서 처리

**학습**: 실시간 데이터 처리 시 스레드 분리와 데이터 버퍼링이 중요

### 도전 3: Pan-Tompkins 알고리즘 구현

**문제**: 논문을 참고하여 구현했지만, 실제 ECG 신호에서 R-peak 검출 정확도가 낮음

**해결**:
- 적응형 임계값 알고리즘 개선
- RR 간격 제약 조건 추가 (생리학적 한계 고려)
- 원본 신호에서 R-peak 위치 재검색 (정제 단계)

**학습**: 알고리즘을 그대로 구현하는 것보다, 실제 데이터에 맞게 튜닝이 필요함

### 도전 4: 네트워크 통신 (Android → Python)

**문제**: 
- `NetworkOnMainThreadException` 발생
- 서버 IP 주소를 하드코딩해야 함

**해결**:
- 네트워크 작업을 별도 스레드에서 수행
- 서버 시작 시 로컬 IP를 자동으로 출력하도록 개선
- Android 앱에서 IP 주소를 쉽게 변경할 수 있도록 주석 추가

**학습**: Android에서는 메인 스레드에서 네트워크 작업을 할 수 없으며, 비동기 처리가 필수

### 도전 5: ECG 서명 비교 정확도

**문제**: 같은 사람의 ECG라도 측정 조건에 따라 서명이 달라져 인증 실패

**해결**:
- 특징 벡터 정규화 강화
- 코사인 유사도 사용 (벡터 방향 비교)
- 임계값을 0.75로 설정 (너무 엄격하지 않게)

**한계**: 실제 사용자 데이터로 더 많은 테스트가 필요함

**학습**: 생체 인증은 변동성이 있으므로, 임계값 설정이 중요하며 실제 데이터로 검증이 필요

### 도전 6: 멀티스레딩 및 동시성

**문제**: 여러 클라이언트가 동시에 접속할 때 데이터가 섞임

**해결**:
- Python 서버에서 각 클라이언트마다 별도 스레드(ClientHandler) 생성
- 각 클라이언트마다 독립적인 ECGProcessor 인스턴스 사용
- 세션 ID로 클라이언트 구분

**학습**: 서버 개발 시 동시성 처리가 중요하며, 스레드 안전성을 고려해야 함

## 6. 예상 질문과 답변

### Q1. 왜 ECG를 선택했나요?

**답변**: 
"기존 생체 인증 방식(지문, 얼굴 인식)의 한계를 보완하고자 ECG를 선택했습니다. 

**장점**:
1. **고유성**: 사람마다 ECG 파형이 다르며, 특히 형태학적 특징(P, QRS, T 파)이 개인마다 고유합니다.
2. **생체 신호**: 위조가 어렵고, 살아있는 사람만 측정 가능합니다.
3. **연구 가치**: ECG 기반 인증은 상대적으로 새로운 분야로, 연구 가치가 높습니다.

**단점**:
- 측정 조건(운동, 스트레스)에 따라 변동성이 있습니다.
- 센서 부착이 필요하여 편의성이 떨어질 수 있습니다.

하지만 의료 기기나 보안이 중요한 환경에서는 충분히 활용 가능하다고 생각합니다."

### Q2. Pan-Tompkins 알고리즘을 왜 사용했나요?

**답변**:
"Pan-Tompkins 알고리즘은 ECG R-peak 검출의 표준 알고리즘입니다.

**선택 이유**:
1. **검증된 알고리즘**: 1985년 제안되어 수십 년간 검증되었습니다.
2. **실시간 처리**: 적분과 적응형 임계값을 사용하여 실시간으로 R-peak를 검출할 수 있습니다.
3. **정확도**: 다양한 노이즈 환경에서도 높은 정확도를 보입니다.

**구현 과정**:
- 논문을 참고하여 6단계로 구현했습니다 (필터링 → 미분 → 제곱 → 적분 → 피크 검출 → 정제).
- 실제 데이터에 맞게 임계값과 RR 간격 제약 조건을 튜닝했습니다.

**대안 고려**:
- 간단한 임계값 기반 방법도 구현했지만, 정확도가 낮아 Pan-Tompkins를 선택했습니다."

### Q3. 인증 정확도는 어느 정도인가요?

**답변**:
"현재는 더미 데이터와 소수의 테스트로만 검증했습니다.

**현재 상태**:
- 더미 데이터: 100% 정확도 (같은 파라미터 사용)
- 실제 데이터: 아직 충분한 테스트가 필요합니다.

**예상 정확도**:
- 논문에 따르면 ECG 기반 인증은 85-95% 정도의 정확도를 보입니다.
- 우리 시스템도 유사한 수준을 기대하지만, 실제 사용자 데이터로 검증이 필요합니다.

**개선 방향**:
- 더 많은 사용자 데이터 수집
- 임계값 최적화 (현재 0.75)
- 특징 가중치 조정
- 머신러닝 모델 적용 고려"

### Q4. 실시간 처리가 가능한가요?

**답변**:
"현재는 배치 처리 방식입니다.

**현재 방식**:
- 10000개 샘플(약 20초)을 모은 후 한 번에 처리
- 등록/로그인 시에만 처리

**실시간 처리 가능 여부**:
- 기술적으로는 가능합니다. Pan-Tompkins 알고리즘은 실시간 처리가 가능합니다.
- 하지만 특징 추출과 서명 생성은 여러 비트가 필요하므로, 최소 3-5초 정도의 데이터가 필요합니다.

**개선 방안**:
- 스트리밍 방식으로 변경 (버퍼에 일정량 쌓이면 처리)
- 슬라이딩 윈도우 방식 적용
- 웹소켓으로 실시간 모니터링 추가"

### Q5. 보안은 어떻게 처리했나요?

**답변**:
"현재는 기본적인 보안만 구현했습니다.

**현재 보안 조치**:
1. **ECG 서명 해시화**: SHA-256 해시를 사용하여 원본 데이터를 저장하지 않습니다.
2. **일방향 함수**: 해시는 복원 불가능하여 프라이버시를 보호합니다.
3. **세션 관리**: 로그인 시 세션 ID를 발급하여 세션 관리합니다.

**부족한 부분**:
1. **전송 암호화**: TCP 통신이 평문으로 전송됩니다 (TLS/SSL 미적용).
2. **저장 암호화**: JSON 파일에 평문으로 저장됩니다.
3. **인증 토큰**: 세션 관리가 단순합니다.

**개선 방향**:
- TLS/SSL 적용
- 사용자 데이터 암호화 저장
- JWT 토큰 기반 인증
- SQLite/PostgreSQL로 데이터베이스 전환"

### Q6. 프로젝트에서 가장 어려웠던 부분은?

**답변**:
"가장 어려웠던 부분은 **Pan-Tompkins 알고리즘 구현과 튜닝**이었습니다.

**어려웠던 이유**:
1. 논문을 참고했지만, 실제 구현 시 세부 파라미터 조정이 필요했습니다.
2. 다양한 노이즈 환경에서도 정확하게 R-peak를 검출해야 했습니다.
3. 적응형 임계값 알고리즘을 구현하는 것이 복잡했습니다.

**해결 과정**:
- 논문을 여러 번 읽고 각 단계를 이해했습니다.
- 실제 ECG 데이터로 테스트하며 파라미터를 조정했습니다.
- RR 간격 제약 조건을 추가하여 생리학적으로 불가능한 피크를 제거했습니다.

**학습한 점**:
- 알고리즘을 그대로 구현하는 것보다, 실제 데이터에 맞게 튜닝하는 것이 중요합니다.
- 신호 처리 분야에서는 파라미터 조정이 성능에 큰 영향을 미칩니다."

### Q7. 프로젝트의 한계점은?

**답변**:
"여러 한계점이 있습니다.

**기술적 한계**:

3. **처리 속도**: 배치 처리 방식으로 실시간 처리가 아닙니다.

**구현상 한계**:
1. **데이터 저장**: JSON 파일 기반으로 대용량 데이터 처리에 한계가 있습니다.

**하드웨어 한계**:
1. **센서 부착**: 편의성이 떨어질 수 있습니다.
2. **측정 환경**: 운동, 스트레스 등에 영향을 받을 수 있습니다.

**개선 방향**:
- 데이터베이스 연동
